不知道为什么起这种知乎式的标题连我自己都有点反感了（其实是文笔差想不出什么更好的标题 : ) ）。现在我已经完成了英国YEAR 2（也就是国内的大三）的课程，结合过去一年学弟学妹学习时的反馈，对自己曾经的学习心得进行了修正。我想我的这些建议对于任何一个想认真学习计算机科学的人都一定会有帮助。是我这段时间的点点滴滴总结出来的一些经验。

如果你是新人……？

如果你是一个完完全全的新手，你没有任何有关计算机的知识背景（包括不会用谷歌/DuckDuckGo或Bing），那么你该怎么办呢？

很简单，找个你信任的人带带你（认真脸），或者和一些已经有些背景又乐于分享的人交交朋友，只要你能忍受一些时不时突然中二的氛围，一般来说能从他们身上受益很多的。当然，什么叫“受益”这个就比较难界定了，有的小伙伴觉得有学到东西就叫受益了，我觉得最好你能在一段时间内逐步拥有自己独立解决问题的能力（当然可以有搜索引擎的帮助），这是坠吼的。

如果你想完全自学，我先说一句，在对任何一个学科没有任何了解的情况下就说自学，可能你连互联网的资源都不会自己找，还得让别人“喂”你，又谈何“自学”呢？

说不定你身边就有“先飞”的同学，你大可以向他／她请教。但前提是你要有足够的批判精神，敢于提问，多怀疑对方的回答。不要害怕质疑他人，你们在谈论的是科学和学习有关的事情，一些细节很重要，关乎你对整个计算机架构的理解和认识。当然，提出质疑时的态度和礼貌怎么把握，就要看你自己了。

我之所以强调“批判精神”的另一个原因是，人都会犯错，不论是比你年长的，还是比你经验丰富的，不论是你是我，都会犯错。作为一个学生，你不应该让偏见充斥你的思维，而是尽可能多尝试，多认识，这样才能给出更正确的答案。

如果你碰巧是那个能回答问题的同学，如果你是一个热心解答他人问题的人，当然会很辛苦，但我相信你在帮助他人的过程中自己也会受益匪浅。

当你有了一些经验之后，就可以自己去上网查资料获取你想了解的信息了。

下面是大二年上学期的课程的学习建议。有关课程更详细的信息可以在eBridge的Programme and Module 下的 Programme and Module Specifications 的子条目 Module Specifications 中查到。

CSE101 - Computer Systems

这门课叫计算机系统。当时给我们这届上课的是一位有些白发的高个子教授。课程涉及到的知识主要可以用三本书来概括：计算机系统结构（原书第2版），计算机组成：结构化方法以及汇编语言：基于x86处理器（原书第7版）。其中第一本《计算机系统结构（CSA）》是本课程所使用的课本。不过本课本的作者写作水平实在是够烂——导致这本书翻译出来也很烂。作者在这本给大一学生的书中经常给出不带注解又查不到的专业名词，很明显是基于自己直觉和印象又不加校验写出来的。你只能凭自己的直觉和这个名字的构成去猜他大概是个什么东西。出于这样的原因，理解作者的语言变得很困难，减缓了你理解的节奏。我怀疑拿这本书当课本的老师到底有没有仔细读过这本书……？

出于这样的原因，你就得读学校推荐阅读给出的塔嫩鲍姆著的《计算机组成：结构化方法》，这是一本很出色的书，讨论的内容同样也是Computer Systems，但却在语言和内容上比CSA厚实很多很多。也许你在读这本书的时候会对学校的这一书本设置感到奇怪：为什么《CSA》居然是我们的课本，而推荐阅读的书却比课本（在写作上）好上这么多倍（又也许，是塔嫩鲍姆是语言功底比《CSA》作者出色点吧）！我在这里推荐有时间的你直接去看《计算机组成》这本书。往年部分习题涉及的考点，比如脏页（dirty pages）还有分页这些东西，在《计算机组成》这本书里都有很详细的解释，而在《CSA》里却没有，谁知道是为什么呢……可能选用这本教材只是为了和利物浦大学实现教材同步吧……

其实这也是个很有说服力的例子，多读书总是好的。课本并不是唯一获取知识的途径，而如果要了解更多知识，延伸的阅读材料也是十分重要的。

这门课还算是比较值得听的，是我当时除了EAP外翘课最少的课程了。比起CSE002这门课（CS的导论课）却更有计算机科学导论课的味道（这主要是因为我那时候CSE002覆盖的内容太杂吧。虽然覆盖到了计算机科学的很多方面，计算机科学的魅力却没有被很好地展示出来，如果我没记错的话你们结尾学的是Processing，Processing本身是很有意思的，感兴趣的同学也可以在假期了解一下这门语言）。需要注意的是关圣威的声音很小，只有尽可能地去前排才能听到他在讲什么名堂。

如果你没有自己去过一遍PPT的觉悟，还是听课吧。每章PPT的小节最后都有一些习题，其中一部分习题讲师会在课上公布答案。还有几节课是专门的习题课。这些习题都很重要（仅仅说是对考试而言，会在期末考中出现不少仅改过变量的原题）。小题不难，有认真学习的一定会做，没有几个要死记硬背的知识点。而且考试大题内容的花样大同小异，做一做往年的卷子其实就可大部明了会考什么了。如果你有什么不明白的概念，先看PPT，再看课本，如果课本还没有找到或者还不能理解，请看塔嫩鲍姆的《计算机组成：结构化方法》。

由于这门课程还涉及一定量的C语言的内联汇编（即inline assembly，也就是在C语言中嵌入汇编语言来编程），而且有大量的内容要求你自学，因此有一本汇编指令的手册在身边是十分方便的，至于我这句话为什么这么说，等你到时候要准备写作业了就知道了。在这里推荐《汇编语言：基于x86处理器》。这本书不需要一点一点的慢慢看，把它当工具书看效果也很不错。如果你对C语言还没有任何了解，建议可以去看看一些网课或者书籍。我当时看的是The GNU C Programming Tutorial，内容短小精悍，不浪费时间在多余的事情上，还方便上手，但实操的前提是你有过使用编译器或者像Visual Studio这样的IDE（Integrated Development Environment）的基础。当然，你也可以在同时参考电气学生上的C语言课程使用的课件。上这门课如果你能找一个比较熟悉计算机或者C语言的同学／老师（尤其助教）让他／她给你讲解一下课件的内容，你会上手得更快。

关于汇编的部分，我的一位朋友Track有跟我推荐过Human Resource Machine这个游戏，我自己也体验过，对要熟悉汇编的新手来说，确实不错，简单来说这个游戏做的事情就是用类似汇编的语言来写算法（这令我想起Knuth的TAOCP，也许你知道我在说什么……），对写汇编程序的思维提升是有一定帮助的。如果你完全不熟悉汇编，也可以试试从这个游戏上手。

在课程涉及到逻辑电路的部分可以使用逻辑电路的模拟软件Logisim做做实验，动手总是比较有意思的。有关逻辑电路的部分其实也自己可以再看一些书多了解一下，EE下有一门专门介绍逻辑电路的课程，由于我个人这方面接触的也不多，就不多说了。

另外再说一本值得看的书：编码：隐匿在计算机软硬件背后的语言。这本书的前半部分是挺不错的，它从摩斯电码开始，慢慢讲起他眼中计算机的发展史。学习这些背景知识对你了解计算机架构还是比较有用的。虽然本书在某些地方还有些差强人意，但它已经是十分友好的一本书了。我感觉前半部分甚至可以作为非计算机科学学生的读物，还是挺精彩的。

关于学习C语言

编程离不开实践，你必须要自己打出书中的代码才有写某一门语言的感觉。在这样的条件考量下，如果你还有比较多的时间，选择介绍比较全面而专业的Stephen Prata的C Primer Plus也是还可以的。我本人不觉得在这个阶段C语言值得花太多时间去深入学习，除非你感兴趣IoT（Internet of Things，物联网）。

C语言是有很多历史糟粕的，随着语言的更新，有一部分问题得以修复，但语言的更新却不意味着多数程序员用语习惯和硬件上及时的更新。比如，到现在教C语言的时候大家还是在用1和0来当作布尔值（boolean）来用。这种做法从语义是来说是错误的，然而对于业界却是可以接受的，毕竟这种做法有很长的历史了。一门糟糕设计的语言，可能会让你保留当时为了克服语言问题而培养出的的种种恶习。

我不推荐慕课网、计蒜课。不论是从他的习题设计还是教材设计，内容对与计算机科学学生来说都是不合格的。当然，我这么说是希望你能学到最多的东西，如果你想要学到最多，就尽量不要看中文教材甚至中文翻译版的教材（一点儿也没错）。国内目前普遍对于计算机科学的教育没有特别严谨的态度，当然，你要是能找到国内一些院校出的好书，也是可以的。

CSE103 - Introduction to Database Systems

这是门数据库系统导论课程，我当时的主讲是一位没什么头发的、有点驼背习惯的讲师。课本是数据库系统：设计、实现与管理（基础篇）（原书第6版）1。书本的内容上是比较丰富的，例子也还可以，要多花时间读（最好跟着讲师发课本章节的进度看吧，如果出了作业再看就有点太着急了）。但不论是数据库设计还是关于SQL的部分，都要自己多练习练习。经验告诉我，解决实际问题和看书完全是两回事（事实上这个说法对任何课程都适用）。这门课提供的实践偏少，课本里的概念往往在一些地方过于简化，在另一些地方过于复杂化了，因此如果要充分利用这门课，就需要你去课外寻找资源学习了。我可以很负责任地说，在业界需要解决的问题和你在这门课的作业考试里需要解决的问题完全不一样。我建议多了解了解数据库设计的一些案例（case），有机会的话学习一下如何对数据、信息进行建模，这样的知识不光是对这门课很有帮助，或许还能提升你对编程的理解能力（理解编程是围绕信息展开的计算建模）。我在这里推荐佐治亚理工发布在Udacity上的数据库课程。

本课程会涉及MySQL的知识，但不用急着在假期看。

这门课的作业每年好像都差不多（误）

考试中涉及很多书（课本）中的概念（他的课件，以及他上课讲到的概念实在太少，因此有很多差评啊）。比较坑的是，这门课居然会考你什么公司的谁谁谁干了啥，在什么年份干了啥这样的问题……所以包括讲师放到ICE上的一些相关话题的PDF，尽可能都要看一遍，还要记住那些人名和事件。总之，讲师的这门课，你把他发的所有课件（当然包括书）都好好看一遍准没错。考试前还会在ICE上发一些题目，这些题和考试有至关重要的联系，一定要好好找答案。

CSE105 - Introduction to Programming in Java

CSE105的讲师在2017年进行了更换，原因据说是我那届的老师因为觉得苏州空气太差离开了。我对这位新讲师虽然并没有很多接触，但我好歹也跟着学弟学妹又做了一年的作业了，这个短裤博士教的一些优点和缺点，我心里都稍微有数。他敢于开刀，大改了课件，试图从一些比较有意思的例子出发上手编程，比如图形界面编程（GUI），也开始教你用一些Java SE8的新特性，新API了。遗憾的是，他的改变收效甚微。下面我简单介绍一下他课程的风格和问题，下面的一些内容可能你需要学习了一些Java才能理解。

在新设计的课程里，短裤博士把Swing作为入门Java的第一步。然而Swing编程有太多convention（在历史影响下产生的习惯需求），条条框框非常多。对于连基本表达式都不知道怎么写的学生来说，从Swing开始学习编程显然不是什么很聪明的做法。因此，在开头阶段你可能会有点难受。如果你是完全的编程新人，我建议你可以不管他，先从别的地方学一下怎么用Java怎么编程非GUI的一些简单的程序（比如看官方的Java Tutorial），然后再回头去看Swing的内容。只有你理解了语言的语义和语法，你才知道你在干嘛，否则不就是在黑暗里乱撞？我支持亲自试验来学习语言的使用方法，然而这句话只适用于有一些编程常识的同学，并不适合所有人。当然，如果你以前学过Python，Pascal，C++，Typescript，Ruby，Scheme……等等有面向对象编程特性的语言，那么，你可以试试从Swing这个框架开始理解Java的语法和语义是怎样的。从Java 9开始，JDK已经有自带JShell作为Java的REPL，你可以用它来试验Java。

除此以外，短裤博士过分重视了Interface在实际编程中的重要性。Java语言的一个重要抽象设施就是Interface，然而，过度强调Interface反而会让学生在实际编程过程中过早抽象。学生可能不清楚在什么时候该用Interface，什么时候该用Abstract Class，什么时候都不需要用，对抽象的时机和需求没有直觉。这一点你需要在正式上课的时候注意。

关于编程的其他建议

简单来说，学习编程并不只是单纯的为了设计能实现某个功能的算法，更重要的是要学会如何建模以控制工程复杂度。

在你学习编程的过程中，也要同时注意语言的哪些特性、语法是好的，是可以反复用的，哪些特性或语法在使用的时候有限制，会带来额外的开销。

举个例子，Java中的 if 表达式模仿了C语言的写法：

if (condition) {
   expressions...
}

有的人会把看错代码这种事情怪就到一个人的水平。的确，富有经验的观察者可以根据语境和程序的意图一定程度上判断什么地方应该写什么代码。但看错了这种代码就能说明某一个人的“水平”低吗？这本来就是写代码的人造成的过失：他自以为这种写法优雅简单，却没有遵循最好的编程实践，浪费了他人读他代码的时间。

假设条件为foo（foo ==> true），函数体的内容为表达式foo = false;（将foo变量的值改变为false），那么这个if语句可以写成。

if (foo) {
   foo = false;
}

Java和C都允许你在body部分只有一条语句时省略这个语句的花括号:

if (foo) foo = false;

看似方便，“清爽”了很多，可这并不见得是一个好的编程习惯，因为它增加了视觉上的开销，读程序的程序员需要反应一下才知道if语句的body在什么位置。使用花括号的时候，if语句的body部分就一目了然了。

这一点你在学习下半学期的CSE104时或许可以感受到这种一时爽省略括号的写法是多糟糕，这种写法甚至能让你看错课件。

有的人会把看错代码这种事情怪就到一个人的水平。的确，富有经验的观察者可以根据语境和程序的意图一定程度上判断什么地方应该写什么代码。但看错了这种代码就能说明某一个人的“水平”低吗？在很多情况下，这是写代码的人造成的过失：他自以为这种写法优雅简单，却没有遵循最好的工程实践，浪费了他人读他代码的时间。工程是一个团队的事情，而每一个细节却都可能影响到整个团队的进度。

这些是需要时间的沉淀来积累的。然而，即使能够掌握这些，也并不意味着你真正掌握了这门语言的全部（全部不代表“框架”）。甚至是最核心的概念OO（Object-Oriented，面向对象）很多同学在上这门课的时候都一直不能理解到底是怎么回事。你应该尝试去搞懂这些非常基本的问题：为什么Java是“面向对象编程”？“面向对象”的特性体现在这门语言的哪里？我该怎么进行“面向对象编程”，它有一定的模版或者模式吗？面向对象编程和传统的过程式编程（如C）有什么不一样的地方？Java有哪些优点或者缺点？为什么C++不设计垃圾回收（Garbage Collection）？等等等等。这些都是十分值得你去思考的问题。

不要盲目成为一门语言的拥护者，也不要盲目鄙视某一门语言：尽管有“图灵完备（Turing Complete）”这么一说，不代表学习别的语言没有价值，更不代表所有语言都是相同的。如果你是ICS的学生，那么在往后的日子，你还会接触到像Prolog这样的逻辑式编程语言，届时你会发现这些语言是如此不同！你要注意到一门语言好的地方和它不好的地方，想想为什么它会在这个语言里好，为什么在另外一个语言中又呈现不一样的状态，做到了这些后，你才会对编程有一些比较深刻的认识。

另外，我想要强调一句，不要因为Java的编程课简单就花最少时间。除非你的生活确实忙不过来，否则这种想法很蠢。几乎每一个计算机科学的东西，都离不开编程语言的介入。你应该多花时间多看看你自己的代码，问问：我代码的质量如何？它好理解吗？它的性能（效率）怎么样？为什么别的同学是那么写的？还有什么能改进的地方？学会在适当的地方做抽象，不断尝试“精炼”、琢磨你的代码。你的技术一定会在这个过程中不断进步。

Java的编程课（也就是CSE105）很有可能是整个大二学年给分最仁慈的一门课。但我真心希望你能多花些时间，去想想怎么构筑你的代码：因为编程真的是计算机科学的精华之一，只有你掌握了编程，你才能慢慢打开通往计算机科学不同道路的门。

这门课没有课本，但是以下资源很不错，可供参考：

  • Java语言导学（原书第5版）2
  • Java语言程序设计（基础篇）（原书第10版）
  • How to Design Programs

CSE107 Foundations of Computing Science

离散数学可怕吗？不可怕。离散数学重要吗？很重要！由于后者的重要性我领悟到得太晚，我在这里由衷地希望大家好好学习离散数学、逻辑学！巩固好微积分、线性代数的知识！要自己多看、多钻研。乍看起来学计算机好像编程不怎么需要用到数学。错了，编程可能用不到多少数学，然而计算机科学，是需要你的数学底子的（这还不算一些小方向可能会涉及到统计学、抽象代数……）。

课件和书本是这门课的灵魂，如果有不懂的地方要多看看题目，做多了，多看看格式，就会答题了。不要以为计算机科学数学不重要，这种认识是完全错误的！

顺便说一下这门课的课本Discrete mathematics for computing / Rod Haggarty 很难借到，需要的话要赶紧下手了。

这门课的讲师好像是意大利海归吧，满身热情洋溢，也是少数几个比较负责任的好老师了。

如果你不喜欢被点名，记住上这门课别穿黄色的衣服！
